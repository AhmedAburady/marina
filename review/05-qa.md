## Findings

### Zero automated tests across the entire project
- **Severity**: P1
- **Category**: qa
- **Location**: repo-wide; `commands/*.go`, `internal/*/*.go`
- **Evidence**: `find . -name '*_test.go'` returns only skill-example files. CLAUDE.md itself says "no test suite exists yet".
- **Why it matters**: Marina fans out mutating docker/compose operations to N homelab hosts over SSH. Every correctness guarantee (digest comparison, stack grouping, partial-failure semantics, state-cache fallback) is enforced only by human review. A single wrong refactor in `internal/registry/check.go` or `internal/discovery/discovery.go` ships straight to users. Go 1.24's `testing/synctest` was designed exactly for deterministic tests over fan-out goroutines and timers — Marina is the canonical use case.
- **Recommendation**: Start with highest-ROI unit seams, then layer on integration:
  1. **`internal/registry` tests with fake registry** — `go-containerregistry` ships `pkg/registry` which starts an in-process registry over `httptest.Server`. Cover: up-to-date, update-available, `@sha256:` pinned short-circuit, non-running containers skipped, in-cycle dedup via `sync.Map`.
  2. **`internal/discovery` table-driven tests** — `GroupByStack` is a pure function over `[]types.Container`; easy golden cases for running-first ordering, stopped stacks merged, missing labels.
  3. **`internal/config` round-trip tests** — YAML load/save/override precedence; uses `t.TempDir()`.
  4. **`internal/ssh` mock server tests** — spin up `golang.org/x/crypto/ssh` server on `127.0.0.1:0`, cover `Exec` stdout+stderr concatenation, `Stream` line pipelining, known_hosts rejection path.
  5. **`internal/actions.FetchAllHosts` fan-out tests** — wrap in `synctest.Run` (Go 1.24+) to get deterministic goroutine scheduling; assert partial-failure behavior with one slow host.
  6. **CLI golden tests** — `commands/ps.go` output under `ui.PrintContainerTable` compared against `testdata/*.golden` with a `-update` flag, so scripted consumers see breakage in CI rather than in prod.
- **Effort**: L (staged: S for registry+discovery first, M for ssh+actions, L for TUI)

### `go.mod` declares 1.26.2 but workflow pins floating `'1.26'`
- **Severity**: P1
- **Category**: qa
- **Location**: `.github/workflows/release.yml:21`
- **Evidence**:
  ```yaml
  - name: Setup Go
    uses: actions/setup-go@v5
    with:
      go-version: '1.26'
  ```
- **Why it matters**: `go.mod` requires `go 1.26.2`; the CI setup resolves `'1.26'` to the latest 1.26.x at build time. Until Go 1.26.2 is released, CI will silently fail (or use 1.26.0 if toolchain directive auto-downloads). When a future 1.26.x patch introduces a regression, reproducibility is lost — two tagged releases built days apart can ship different compiler output. Go 1.26 specifically reworked the GC (Green Tea enabled by default) and heap base randomization; drift between patch versions matters.
- **Recommendation**: Pin to `go-version-file: go.mod` so the workflow always matches the repo's declared version:
  ```yaml
  with:
    go-version-file: go.mod
    check-latest: true
  ```
- **Effort**: S

### Release build missing `-trimpath` — non-reproducible binaries leak absolute paths
- **Severity**: P1
- **Category**: qa
- **Location**: `.github/workflows/release.yml:29-38`
- **Evidence**:
  ```
  go build -ldflags "$LDFLAGS" -o marina-linux-amd64 ./cmd/marina
  ```
- **Why it matters**: Without `-trimpath`, the compiled binary embeds the runner's absolute `$HOME`/`$GOPATH` paths in panic stack traces and DWARF debug info (e.g. `/home/runner/work/marina/marina/internal/ssh/ssh.go`). Two consequences: (1) binaries are not bit-identical across CI runs, breaking reproducible-build verification that downstream distros (Homebrew, Arch AUR, nixpkgs) increasingly require; (2) leaks internal directory structure to anyone who reads `strings marina-linux-amd64`. CLAUDE.md already specifies the build invocation without `-trimpath` either — propagate the fix to both.
- **Recommendation**:
  ```bash
  go build -trimpath -ldflags "$LDFLAGS" -o ...
  ```
  Also update the example in `CLAUDE.md`.
- **Effort**: S

### Release workflow publishes artifacts with no checksums or signatures
- **Severity**: P1
- **Category**: qa
- **Location**: `.github/workflows/release.yml:40-50`
- **Evidence**: `softprops/action-gh-release@v1` uploads raw binaries only; no `SHA256SUMS`, no `cosign` signing, no SLSA provenance.
- **Why it matters**: Marina is downloaded and executed with root-adjacent privileges (SSH + Docker). Users installing via `curl | sh` have no way to verify integrity. Homebrew formulas require a SHA256; without a published checksum file, every downstream packager has to compute one manually and hope the binary wasn't tampered with mid-flight.
- **Recommendation**: Add a step before the release action:
  ```yaml
  - name: Generate checksums
    run: |
      sha256sum marina-* > SHA256SUMS
  ```
  Then include `SHA256SUMS` in the uploaded files. For extra credit, switch to `goreleaser` — it handles checksums, archive packing (`.tar.gz` / `.zip` per OS), and cosign signing with a single config file, replacing the ~30 lines of handwritten build steps.
- **Effort**: S (manual) / M (full goreleaser migration)

### Release workflow never runs tests or `go vet` before publishing
- **Severity**: P1
- **Category**: qa
- **Location**: `.github/workflows/release.yml:11-38`
- **Evidence**: Workflow jumps from `checkout` → `setup-go` → `build binaries`. No `go test`, no `go vet`, no `golangci-lint`.
- **Why it matters**: A tag push with a broken `main` ships a broken binary. Once tests exist (prior finding), this gate becomes essential.
- **Recommendation**: Gate the release job on a `test` job:
  ```yaml
  test:
    runs-on: ubuntu-latest
    steps: [checkout, setup-go, run: go test -race ./...]
  build:
    needs: test
    ...
  ```
- **Effort**: S

### No CI on push/PR — only release triggers a build
- **Severity**: P1
- **Category**: qa
- **Location**: `.github/workflows/` (only `release.yml` exists)
- **Evidence**: Workflow triggers solely on `push: tags: v*`. No `pull_request` or `push: branches: [main]` workflow.
- **Why it matters**: Every commit lands on `main` unvetted by CI. Contributors (and Claude-driven edits) can merge compilation failures. This is the foundational gap — even the existing lack of tests can't be enforced without CI.
- **Recommendation**: Add `.github/workflows/ci.yml` with matrix `os: [ubuntu-latest, macos-latest, windows-latest]` running `go build ./...`, `go vet ./...`, `go test -race ./...`, and `golangci-lint run` on every PR and push to `main`.
- **Effort**: S

### Build matrix never tests darwin/windows — cross-compile ≠ validation
- **Severity**: P1
- **Category**: qa
- **Location**: `.github/workflows/release.yml:13` (`runs-on: ubuntu-latest`)
- **Evidence**: All six targets cross-compiled from a single Linux runner. No `macos-latest` or `windows-latest` job ever exercises the binary.
- **Why it matters**: Cross-compiling only proves the code *compiles* for GOOS=darwin — not that it *runs*. The SSH known_hosts path (`internal/ssh/ssh.go:74`), the `~/.config/marina/` hardcoding, and `commands/root.go`'s TTY detection all contain paths that diverge per OS. Marina advertises six targets; it should validate at least three.
- **Recommendation**: Run the test suite (once it exists) on all three OS runners. Keep the release build on `ubuntu-latest` for reproducibility but add a matrix CI job:
  ```yaml
  strategy:
    matrix:
      os: [ubuntu-latest, macos-latest, windows-latest]
  runs-on: ${{ matrix.os }}
  ```
- **Effort**: S

### Hardcoded `~/.config/marina/` breaks XDG and Windows conventions
- **Severity**: P2
- **Category**: qa
- **Location**:
  - `internal/config/config.go:83`
  - `internal/state/state.go:19`
  - `internal/registry/cache.go:38`
  - `internal/tui/log.go:42-48`
- **Evidence**:
  ```go
  return filepath.Join(home, ".config", "marina", "config.yaml"), nil
  ```
- **Why it matters**: On Linux this ignores `XDG_CONFIG_HOME`, on macOS it diverges from `~/Library/Application Support/marina`, on Windows it creates `%USERPROFILE%\.config\marina\` instead of the canonical `%AppData%\marina\`. `os.UserConfigDir()` (stdlib since Go 1.13) returns the right thing on all six target platforms, and `os.UserCacheDir()` is the correct home for the `check-cache.json` (it is a cache, not config).
- **Recommendation**: Replace the four call sites with `os.UserConfigDir()` / `os.UserCacheDir()`. Keep a back-compat read-fallback for one release cycle so existing users don't lose their config:
  ```go
  cfgDir, err := os.UserConfigDir()
  if err != nil { return "", err }
  return filepath.Join(cfgDir, "marina", "config.yaml"), nil
  ```
- **Effort**: M (four files + migration note)

### SSH agent auth silently no-ops on Windows
- **Severity**: P2
- **Category**: qa
- **Location**: `internal/ssh/ssh.go:91-100`
- **Evidence**:
  ```go
  if sock := os.Getenv("SSH_AUTH_SOCK"); sock != "" {
      agentConn, err := net.Dial("unix", sock)
  ```
- **Why it matters**: Windows OpenSSH agent exposes a **named pipe** (`\\.\pipe\openssh-ssh-agent`), not a unix socket; the Pageant / 1Password agent use named pipes too. `SSH_AUTH_SOCK` is typically unset on Windows. Result: Windows users always fall through to key-file auth with no indication that agent support exists — and since Marina is a cross-platform release, this is a supported config path that is quietly broken.
- **Recommendation**: Add a Windows-only agent dialer behind build tags:
  ```go
  // ssh_agent_windows.go
  //go:build windows
  func dialAgent() (net.Conn, error) {
      return winio.DialPipe(`\\.\pipe\openssh-ssh-agent`, nil)
  }
  ```
  (`github.com/Microsoft/go-winio` is already an indirect dep via docker/docker.) Add a fallback chain: `SSH_AUTH_SOCK` → named pipe → key file. Document in README.
- **Effort**: M

### No golden-file tests for CLI table output — silent breakage risk for scripted consumers
- **Severity**: P2
- **Category**: qa
- **Location**: `internal/ui/table.go` (220 LOC, `PrintContainerTable` + `PrintStackTable`)
- **Evidence**: No `testdata/` directory anywhere in the repo; no golden-file tests in `internal/ui/`.
- **Why it matters**: CLAUDE.md states explicitly: "subcommands print bordered tables (scriptable)". The moment someone scripts `marina ps -H host | awk '{print $1}'` downstream, lipgloss's padding rules become a public API — yet any edit to `table.go`'s columns / borders can silently change output.
- **Recommendation**: Use a snapshot approach — `testing` + `testdata/*.golden` files with a `-update` flag:
  ```go
  func TestContainerTable_Golden(t *testing.T) {
      var buf bytes.Buffer
      PrintContainerTable(&buf, fixtureContainers())
      got := buf.Bytes()
      golden := filepath.Join("testdata", t.Name()+".golden")
      if *update { os.WriteFile(golden, got, 0o644) }
      want, _ := os.ReadFile(golden)
      if !bytes.Equal(got, want) { t.Errorf("output drift:\n%s", cmp.Diff(string(want), string(got))) }
  }
  ```
- **Effort**: M

### TUI has no `teatest` harness — an 8,700-LOC Bubble Tea app with zero regression coverage
- **Severity**: P2
- **Category**: qa
- **Location**: `internal/tui/` (dashboard, home, hosts, containers, stacks, updates, prune, filter, confirm, overlay) — 11 screen files
- **Evidence**: `internal/tui/*_test.go` → 0 files.
- **Why it matters**: Bubble Tea v2 ships `charm.land/bubbletea/v2/teatest` specifically for TUI regression testing (see `bubbletea-v2` skill: `teatest` availability). It drives a `tea.Program` with scripted `KeyPressMsg` inputs and snapshots the final frame. Marina's screen-stack navigation + shared `filterBar` + `confirmPrompt` is exactly the kind of composition where a subtle `Update()` return in one screen can break navigation in another — no tests means the audit log is currently the only regression signal.
- **Recommendation**: Start with smoke tests for the screen interface contract: each screen's `Init`/`Update`/`View` called with a canonical set of messages; assert no panic and `View()` returns non-empty. Then add a few end-to-end teatest scripts (dashboard → hosts → confirm delete → ESC back). Don't aim for exhaustive coverage — aim for detecting the "apply did nothing" class of bugs CLAUDE.md mentions.
- **Effort**: L

### GitHub Actions pinned by floating tag, not SHA — supply-chain risk
- **Severity**: P2
- **Category**: qa
- **Location**: `.github/workflows/release.yml:16,19,41`
- **Evidence**:
  ```yaml
  uses: actions/checkout@v4
  uses: actions/setup-go@v5
  uses: softprops/action-gh-release@v1
  ```
- **Why it matters**: A compromised maintainer of `softprops/action-gh-release` could re-point `v1` at a malicious commit; the next tagged release of Marina would run that code with `contents: write` permission — enough to rewrite the release binary post-build. This is not hypothetical: the `tj-actions/changed-files` compromise (Mar 2025) worked this way. `softprops/action-gh-release` is third-party; pin it at minimum.
- **Recommendation**: Pin all actions by full commit SHA with a version comment:
  ```yaml
  uses: softprops/action-gh-release@<sha>  # v2.0.x
  ```
  Use Dependabot's `package-ecosystem: github-actions` to keep them current. First-party `actions/*` pinning is best-practice; third-party pinning is the non-negotiable piece.
- **Effort**: S

- **Severity**: P3

### Opportunity: `testing/synctest` is the right hammer for Marina's fan-out tests
- **Severity**: P2
- **Category**: qa
- **Location**: `internal/actions/` fan-out logic (referenced by CLAUDE.md as `FetchAllHosts`, `Prune`, etc.)
- **Evidence**: Marina's hot path is concurrent SSH fan-out across N hosts with per-host timeouts. Deterministic tests of "host B times out, host A succeeds, result map is partial" are typically flaky under wall-clock timing.
- **Why it matters**: `testing/synctest` (Go 1.24+, stabilized 1.25) provides a fake clock bubble where `time.Sleep`, `time.After`, and `context.WithTimeout` advance virtually. This makes partial-failure tests deterministic with zero real wall-clock wait. Without it, any concurrent test ends up using `time.Sleep(100*time.Millisecond)` which is both slow and flaky.
- **Recommendation**: When writing the fan-out tests above, use:
  ```go
  func TestFetchAllHosts_PartialFailure(t *testing.T) {
      synctest.Run(func() {
          ctx, cancel := context.WithTimeout(t.Context(), time.Second)
          defer cancel()
          // fake hosts: one slow, one fast
          got := FetchAllHosts(ctx, ...)
          // assertions run at virtual-time t=1s deterministically
      })
  }
  ```
  Also use `t.Context()` (Go 1.24+) which auto-cancels on test cleanup.
- **Effort**: M (pair with the test-writing effort above)

### Opportunity: Go 1.26 `errors.AsType[T]` is a type-safe drop-in for `errors.As`
- **Severity**: P3
- **Category**: qa
- **Location**: any future error-type unwrapping code
- **Evidence**: Go 1.26 release notes: "The new `AsType` function is a generic version of `As`. It is type-safe, faster, and, in most cases, easier to use."
- **Why it matters**: Reviewers should prefer `errors.AsType[*ssh.ExitError](err)` over the older two-step `var e *ssh.ExitError; errors.As(err, &e)` pattern. Relevant once SSH error-type branching is added (it isn't today in `internal/ssh/ssh.go`).
- **Recommendation**: Note in CLAUDE.md's future error-handling section.
- **Effort**: S
